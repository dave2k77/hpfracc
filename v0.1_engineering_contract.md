## 1. Product Position

HPFRACC v0.1 is a research-support library first and a publishable open-source framework second. The initial release should establish a credible numerical and differentiable foundation for fractional calculus and fractional dynamical systems, without attempting to complete the later phantom-brain application stack.

The v0.1 priority order is:

1. Numerical correctness.
2. Numerical stability.
3. Differentiability.
4. Reproducibility and provenance.
5. Performance on CPU and single accelerator.
6. Extensibility toward neural mass, neural field, and phantom-brain models.

The library is research software. It is not clinical software, not diagnostic software, and not a substitute for empirical biomedical validation.

## 2. Scope for v0.1

### 2.1 In Scope

HPFRACC v0.1 shall provide:

- A JAX-native fractional calculus core.
- First-class support for Riemann-Liouville, Caputo, and Grunwald-Letnikov operator families.
- A fixed-step deterministic fractional differential equation solver, with Caputo dynamics as the first solver target.
- Differentiable operator and solver paths where mathematically and computationally defensible.
- Validation tests against analytic identities, limiting cases, polynomial functions, and reference numerical solutions.
- Structured configuration and result objects.
- Explicit random-key and state-passing conventions, even where v0.1 functionality is deterministic.
- CPU and single GPU/TPU execution targets.
- Public API namespaces prepared for future SciML, probabilistic, and brain-model layers.
- Documentation of mathematical definitions, assumptions, shape conventions, differentiability support, and known failure modes.

### 2.2 Out of Scope for v0.1

HPFRACC v0.1 shall not attempt to deliver:

- Full phantom-brain simulation.
- Production-ready neural mass or neural field model libraries.
- Multi-GPU or multi-TPU scaling.
- Complete probabilistic programming workflows.
- Clinical, diagnostic, or patient-specific claims.
- Broad backend support beyond JAX.
- Every known fractional operator or discretisation.
- A large public API surface.

Neural mass, neural field, probabilistic inference, and phantom-brain modules may exist as placeholders or experimental prototypes, but they should not drive the architecture of the numerical core.

## 3. Backend and Dependency Decisions

### 3.1 Numerical Backend

v0.1 shall be JAX-only.

Required implications:

- Arrays should be JAX arrays or JAX-compatible pytrees.
- Public numerical functions should be compatible with `jax.jit` where feasible.
- Public differentiable functions should be compatible with `jax.grad`, `jax.jvp`, or `jax.vjp` where declared.
- Randomness must use explicit JAX PRNG keys for stochastic functionality.
- Python-side mutation and hidden global state should be avoided in public numerical paths.

### 3.2 Accelerator Target

v0.1 shall target:

- CPU.
- Single GPU.
- Single TPU, where JAX supports the workload.

Multi-device support is a future design target. v0.1 APIs should avoid choices that block later `pmap`, `pjit`, or sharding support, but multi-device implementation is not required.

### 3.3 Likely Dependency Policy

Core runtime dependencies should remain small. Candidate dependency tiers:

- Required core: JAX, NumPy-compatible typing utilities, packaging metadata.
- Optional test/validation: SciPy, mpmath, pytest, hypothesis.
- Optional future SciML: Equinox or pure pytree-compatible local abstractions.
- Optional future probabilistic layer: NumPyro.
- Optional docs: MkDocs or Sphinx.

No dependency should be added to the core runtime unless it is necessary for the numerical foundation.

## 4. Public API Contract for v0.1

The canonical import shall be:

```python
import hpfracc as hp
```

The v0.1 top-level namespaces should be:

- `hp.ops`
- `hp.solvers`
- `hp.typing`
- `hp.config`
- `hp.metrics`
- `hp.experimental`

The following namespaces may be present as placeholders or experimental surfaces:

- `hp.nn`
- `hp.prob`
- `hp.brain`
- `hp.observe`
- `hp.train`
- `hp.data`
- `hp.viz`

Stable v0.1 APIs should be minimal. Most public objects should begin as provisional until numerical validation and user workflows settle.

## 5. Operator Requirements

v0.1 shall focus on three operator families:

1. Riemann-Liouville operators.
2. Caputo operators.
3. Grunwald-Letnikov operators.

### 5.1 Required Operator Capabilities

Each implemented operator shall document:

- Mathematical definition.
- Domain assumptions.
- Boundary and initial-history conventions.
- Supported fractional-order range.
- Input and output shape conventions.
- Discretisation method.
- Accuracy expectations.
- Differentiability status.
- Known numerical failure modes.

### 5.2 First Supported Orders

v0.1 should initially support scalar fractional orders in the interval:

```text
0 < alpha < 1
```

Vector-valued orders, per-state orders, per-node orders, and trainable order fields are important future requirements but should not be required for the first implementation path unless the core design makes them inexpensive.

### 5.3 First Supported Grids

v0.1 should initially support uniform time grids.

Nonuniform grids are scientifically important but substantially increase the complexity of operator definitions, solver validation, and performance optimization. They should be treated as a later extension unless a specific v0.1 validation case requires them.

### 5.4 Operator Differentiability

The first differentiability targets shall be:

- Gradients with respect to input signal/state values.
- Gradients with respect to model parameters used inside solver dynamics.
- Gradients with respect to initial state where applicable.

Gradients with respect to fractional order `alpha` are desirable but should be explicitly marked provisional until validated against finite-difference and analytic reference cases.

## 6. Solver Requirements

### 6.1 First Solver Target

The first solver target shall be a fixed-step Caputo FDE solver.

Preferred initial method:

- Predictor-corrector Adams-Bashforth-Moulton style method for Caputo equations.

The solver should prioritize correctness, traceability, and validation over aggressive optimization.

### 6.2 Solver Interface

The preferred public simulation signature is:

```python
simulate(
    model,
    ts,
    *,
    solver,
    initial_state=None,
    params=None,
    rng_key=None,
    inputs=None,
    observe=False,
    config=None,
)
```

The preferred dynamics signature is:

```python
f(t, state, params, *, rng_key=None, inputs=None)
```

or, for model objects that carry parameters:

```python
f(t, state, *, rng_key=None, inputs=None)
```

### 6.3 Solver Result Contract

Simulation routines shall return structured result objects rather than raw tuples.

Minimum fields:

- `ts`
- `latent_state`
- `observed`
- `solver_info`
- `metadata`

The result object should be serializable or exportable to a serializable representation.

### 6.4 Solver Diagnostics

Solver metadata should include:

- Solver name and version.
- Fractional operator definition.
- Fractional order.
- Step size or time-grid summary.
- Number of steps.
- Runtime diagnostics where available.
- Validation warnings where applicable.

## 7. Memory and Performance Strategy

Fractional dynamics are history dependent. v0.1 shall make memory handling explicit rather than hidden.

### 7.1 Initial Memory Strategy

The first implementation may use full-history methods for correctness and validation.

This is acceptable for v0.1 if:

- Complexity is documented.
- Memory scaling is tested.
- Large-problem limitations are clear.
- The API does not prevent later compressed-memory implementations.

### 7.2 Future Memory Strategies

The design should leave room for:

- Short-memory truncation.
- Sum-of-exponentials kernel compression.
- FFT-accelerated convolution.
- Checkpointing/rematerialization.
- Adaptive memory policies.

These should not be silently substituted for exact/full-history behavior. Any approximation should be named and configured explicitly.

## 8. Shape and Batch Semantics

Unless otherwise documented:

- Single state: `(..., state_dim)`
- Simulated trajectory: `(time, ..., state_dim)`
- Observed signal: `(time, ..., obs_dim)`
- Connectivity matrix: `(n_nodes, n_nodes)`
- Node state: `(..., n_nodes, node_state_dim)`

Batch dimensions, when supported, shall be leading non-time dimensions.

Time is introduced as a leading axis only after simulation. Stepwise dynamics operate on a single time-slice state.

## 9. Validation Requirements

Testing is part of the scientific contract, not just a development practice.

### 9.1 Mathematical Validation

v0.1 shall include tests for:

- Constants.
- Polynomials.
- Power-law functions where analytic fractional derivatives are known.
- Limiting behavior as `alpha` approaches classical values.
- Consistency relations between supported operators where applicable.
- Agreement with high-precision or trusted reference calculations.

### 9.2 Solver Validation

v0.1 solver validation shall include:

- Timestep refinement behavior.
- Known analytic or semi-analytic FDE examples.
- Reproducibility under fixed inputs.
- Stability behavior in controlled regimes.
- Failure or warning behavior under invalid parameters.

### 9.3 Differentiability Validation

Declared differentiable paths shall be tested with:

- `jax.grad` smoke tests.
- Finite-difference checks on small examples.
- Shape and pytree compatibility tests.
- JIT and non-JIT consistency checks where practical.

### 9.4 Performance Validation

v0.1 shall record benchmark cases for:

- Compile time.
- Steady-state runtime.
- Memory use.
- Sequence-length scaling.
- State-dimension scaling.
- CPU versus single accelerator behavior where available.

No broad performance claim should be made without benchmark context.

## 10. Stability Tiers

All public objects shall be assigned one of:

- Stable.
- Provisional.
- Experimental.
- Private.

For v0.1:

- Top-level namespace names may be stable.
- Core operator APIs may be provisional.
- Solver APIs may be provisional.
- Neural, probabilistic, brain, observation, and training APIs should be experimental unless fully validated.
- Underscored modules are private.

## 11. Documentation Requirements

v0.1 documentation shall include:

- Installation instructions.
- First operator example.
- First Caputo FDE simulation example.
- Theory notes for RL, Caputo, and GL definitions.
- Shape convention page.
- Differentiability contract page.
- Validation methodology page.
- Research usage and non-clinical disclaimer.
- API reference generated from docstrings or maintained alongside code.

Notebooks may be used for examples, but mathematical definitions and API contracts must live in versioned documentation.

## 12. Research Usage Policy

Release materials shall state that HPFRACC is:

- Research software.
- Not clinical software.
- Not validated for diagnosis, prognosis, or treatment.
- Not a substitute for empirical biomedical assessment.

Published or reported results should record:

- HPFRACC version.
- Commit hash or release tag if available.
- Backend and hardware context.
- Solver/operator configuration.
- Random seed or key lineage.
- Experiment manifest.
- Validation status of operators and solvers used.

## 13. Future Direction After the v0.1 Alpha

After the numerical foundation is credible and the research-ready v0.1 alpha
cleanup is complete, the next development layers should be:

1. Neural mass model primitives.
2. Neural field model primitives.
3. Probabilistic calibration with NumPyro or another explicit optional
   inference backend.
4. More validated stochastic fractional dynamics.
5. EEG observation models.
6. Phantom-brain workflows.
7. Memory compression and multi-device scaling.

The brain stack should be built on top of the validated mathematical core, not
interleaved with it prematurely.

The post-alpha plan that realizes this direction is the Phase A-D roadmap in
`docs/developer/roadmap.md`, which sequences core-hardening ahead of the brain
stack and maps the path from the released v0.1.0a0 kernel to the full layered
blueprint in `hpfracc_design_specification.pdf`. Standing decisions for that plan:
the probabilistic layer stays hand-rolled and NumPyro-free while preserving the
public `hp.prob` surface; the package keeps its flat layout and grows additively
rather than refactoring into the blueprint's granular directory tree.

## 14. Open Questions

The following decisions have been resolved:

1. Package/build flow: `pyproject.toml` with Hatchling, compatible with `uv`.
2. Python version: Python `>=3.11`.
3. Core config/result objects: standard frozen dataclasses; pytree-specific
   registration only where needed later.
4. Operator validation tolerances: case-specific numerical tolerances in tests,
   with validation reports recording raw error values.
5. Gradients with respect to fractional order `alpha`: provisional.
6. SciPy/mpmath: test-only validation dependencies.
7. Experimental additive-noise FSDE support: present after deterministic FDE
   validation; broader stochastic numerics remain provisional.
8. Documentation: MkDocs Material.
9. Experiment manifests: typed Python objects with export support first.
10. First neural mass target: Wilson-Cowan unless later requirements force a
    different first model.

The following questions remain open for later milestones:

1. Which memory compression method should be implemented first after
   full-history solver validation?
2. Which adjoint/checkpointing strategy should be used for larger differentiable
   solver workloads?
3. Which GPU/TPU benchmark environment should define the first accelerator
   baseline?
